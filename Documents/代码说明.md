第一站：Shared 模块（图纸与协议定义）
这部分逻辑不依赖任何手机系统，是跨平台的“灵魂”。
1. Models.kt —— 【协议说明书】
•
它是做什么的：定义了所有数据结构的“形状”。
•
看点：
◦
UiNode (基类)：规定了每个组件必须有 style (皮囊)、props (内容)、state (开关) 和 action (点击行为)。
◦
CornerRadius / Spacing：专门处理圆角和边距的工具模型。
◦
各 Node 子类：现在全都是“空壳”，所有具体数据都塞进了通用的 props 字典里。
2. UiNodeSerializer.kt —— 【数据分拣心脏】
•
它是做什么的：把乱糟糟的原始 JSON 翻译成上面那种规整的 Kotlin 对象。
•
看点：
◦
sanitizeNode (全屋扫描)：它会递归扫描 JSON，自动把老协议（如 styles）修建成新协议（如 style）。
◦
onTap 压扁逻辑：它会让复杂的动作结构变得简单直接。
第二站：App 模块（物理施工队）
这部分负责把“虚拟的协议”画成“Android 的像素”。
1. StyleUtils.kt —— 【物理参数转换器】
•
它是做什么的：把协议里的“原始数字”加上物理单位，并处理对齐。
•
看点：
◦
toPaddingValues / toShape：给边距和圆角自动贴上 .dp。
◦
getBoxAlignment：把 "center" 这种话翻译成 Compose 的坐标对齐。
◦
属性提取器 (getString, getDouble)：安全地从 JSON 口袋里拿出我们要的东西，拿不到就用默认值。
2. ModifierExt.kt —— 【标准化外壳工厂】
•
它是做什么的：给每个组件（Text, Image, Box）穿上一层通用的“紧身衣”。
•
看点：
◦
applyCommonStyle：不管你是文字还是图片，只要你写了 width、height 或 margin，这层逻辑就会自动把你调整到正确的大小和位置。
3. DynamicRenderer.kt —— 【施工现场总监】
•
它是做什么 of：这是最核心的界面，也是你最该反复研读的文件。
•
看点：
◦
分层解析流：
a.
先看 State（画不画？）。
b.
再看 Style（背景色、圆角、宽高等外在装饰）。
c.
最后看 Props（内容是什么？）。
◦
模块化渲染：它把 RenderText、RenderButton 等逻辑拆分成了独立的小房间，互不干扰，代码极其整洁。
◦
递归绘制：如果是容器，它会带着孩子再次回到流水线顶端，直到画完整个页面。
第三站：外部组件库（特种分包商）
1. RollingCounter.kt
•
它是做什么的：专门处理那个酷炫的“老虎机”数字滚动动画。主渲染器只负责调用它。
2. StyleErrorIndicator.kt
•
它是做什么的：你的“调试助手”。如果 JSON 写错了，它就在屏幕右上角画个感叹号，并告诉你错在哪了。
💡 学习建议：
1.
跟踪一个字段：比如在 JSON 里改个 fontSize，看它如何经过 Serializer 解析成 props，再经过 StyleUtils 贴上 sp，最后在 DynamicRenderer 的 RenderText 函数里变成真正的文字大小。
2.
看递归逻辑：研究 RenderBox 是如何调用 DynamicRenderer 本身的，理解 UI 树是如何一层层长出来的。
这套架构不仅是用来做这个弹窗的，它是一套完整的动态 UI 框架。理解了这几个文件，你就能在这个地基上盖出任何大楼！
